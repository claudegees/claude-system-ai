# Architecture Decision Records (ADR)

主要な技術的決定とその理由を記録する。新規参加者が「なぜこうなっているのか」を理解するため。

---

## ADR-001: 認証は OAuth 2.0 + JWT を採用

**日付**: プロジェクト開始時
**ステータス**: 採用中

### 背景

ユーザー認証方式の選定。複数の選択肢を検討:
1. セッション方式（Cookie + サーバ側ストア）
2. OAuth 2.0 + JWT
3. 独自トークン方式

### 決定

OAuth 2.0 + JWT を採用。

### 理由

- ステートレスでスケールしやすい
- モバイルアプリ・SPA・サーバサイドレンダリングすべてに対応
- 業界標準で実装サンプルが豊富
- サードパーティ認証（Google/GitHub等）への拡張が容易

### トレードオフ

- JWTの即時revoke が困難 → 短い有効期限（15分）+ refresh token で対応
- ペイロードサイズが大きくなりがち → 必要最小限の claim のみ含める

## ADR-002: バックエンド言語に TypeScript を採用

**日付**: プロジェクト開始時
**ステータス**: 採用中

### 背景

サーバサイド実装言語の選定。

### 決定

TypeScript (Node.js) を採用。

### 理由

- フロントエンドと同じ言語でフルスタック開発の認知負荷を削減
- 型安全性で実行時エラーを大幅に減らせる
- npm エコシステムが豊富
- 採用しやすい（応募者プールが大きい）

### トレードオフ

- 並行処理性能は Go や Rust より劣る → 重い処理は別マイクロサービス化で対応
- runtime overhead は高め → CPU bound 処理は最小化する設計

## ADR-003: データベースは PostgreSQL を主とし、検索は別途 Elasticsearch

**日付**: プロジェクト開始3ヶ月後
**ステータス**: 採用中

### 背景

検索機能の要件が複雑化。LIKE 検索では対応できないユースケースが増えた。

### 決定

- 一次データソースは PostgreSQL（ACID保証必須）
- 全文検索・ファセット検索は Elasticsearch
- PG → ES への同期はイベントドリブン（CDCツール使用）

### 理由

- データの真実は1箇所に保つ（PG）
- 検索専用ストアで検索性能と機能を両立
- 同期遅延は許容範囲（数秒以内）

### トレードオフ

- 同期の運用コストが追加発生
- 一時的な不整合の可能性 → UI上で「検索は反映に数秒かかる」と明示

## ADR-004: テストは jest + Playwright の二段構え

**日付**: プロジェクト開始6ヶ月後
**ステータス**: 採用中

### 背景

E2E テストツールの選定。Cypress / Playwright / Puppeteer を比較。

### 決定

- Unit / Integration: **jest**
- E2E: **Playwright**

### 理由

- Playwright は複数ブラウザ対応・並列実行が安定
- Cypress は単一ブラウザに制限される時期があった
- jest は React/Node.js コミュニティ標準

### 再検討の予定

なし（特に問題発生していない）
