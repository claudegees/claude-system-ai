# アーキテクチャ概要

## システム構成

```
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│  クライアント │ ───→ │  APIサーバ  │ ───→ │ データベース │
│   (Web/UI)  │      │  (Backend)  │      │  (Primary)  │
└─────────────┘      └─────────────┘      └─────────────┘
                            │
                            ↓
                     ┌─────────────┐
                     │  キャッシュ  │
                     │  (Redis)    │
                     └─────────────┘
```

## レイヤー構造

| レイヤー | 役割 | 主要技術 |
|---|---|---|
| Presentation | UI / API endpoint | Web Framework |
| Application | ユースケース実装 | Service層 |
| Domain | ビジネスロジック | Entity / Value Object |
| Infrastructure | DB / 外部API連携 | Repository実装 |

## 設計原則

- **依存性逆転**: ドメイン層は他層に依存しない
- **単一責任**: 1モジュール1責務
- **境界明示**: モジュール間は型/インターフェース経由
- **副作用の局在化**: I/O は Infrastructure 層のみ

## データフロー

1. クライアントがAPI endpoint を叩く
2. Application層でユースケース実行
3. Domain層でビジネスルール検証
4. Infrastructure層でDB/外部APIアクセス
5. レスポンスを返す

## スケーリング戦略

- **水平スケール**: APIサーバはステートレス、ロードバランサ配下で複数台
- **DB**: Primary/Replica構成、読み取りはreplicaへ
- **キャッシュ**: 読み取り頻度が高いものはRedisに
- **非同期処理**: 重い処理はジョブキューへ
