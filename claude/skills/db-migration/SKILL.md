# Database Migration

DBスキーマの変更を、安全な順序で実行できるマイグレーションとして書く。

## 安全マイグレーションの原則

| 操作 | 安全性 | 注意 |
|---|---|---|
| カラム追加（NULL可） | ✅ 安全 | デフォルト値があれば既存行も埋められる |
| カラム追加（NOT NULL） | ⚠️ 注意 | 既存行のbackfill が必要 |
| カラム削除 | ⚠️ 注意 | アプリの参照を先に消すこと |
| カラム名変更 | 🚫 危険 | 旧→新の同期期間が必要 |
| 型変更 | 🚫 危険 | データ変換のリスク高い |
| インデックス追加 | ✅ 安全 | 大規模テーブルは `CONCURRENTLY` 使う |

## 推奨パターン

### 大きな変更を分割する

```
Step 1: 新カラム追加（NULL可）
Step 2: アプリで両方書き込み（旧と新）
Step 3: backfill スクリプトで既存行を埋める
Step 4: NOT NULL 制約追加
Step 5: アプリを新カラムのみ参照に変更
Step 6: 旧カラム削除
```

→ 各Stepは独立してデプロイ可能

## 手順

1. 現在のスキーマを確認
2. 変更内容と影響範囲を整理
3. 安全な順序に分解
4. up（適用）と down（ロールバック）の両方を書く
5. データ規模を確認し、長時間ロックの可能性を警告

## 守ること

- **down マイグレーションを必ず書く**
- **テストDBで実行確認** してから本番適用
- **大規模テーブルの `ALTER TABLE` は計画的に**: 数百万行で長時間ロックになる可能性
- **PII 削除を含む場合は監査ログ**
